# Free-moN Module Repository

Backend (Supabase/PostgreSQL + Edge Functions) for the Free-moN Module
Repository. See `github.com/willcgage/modulerepo`.

## Project layout

```
.
├── supabase/
│   ├── config.toml              # Supabase CLI project config
│   ├── migrations/              # SQL migrations, applied in filename order
│   └── functions/               # Edge Functions (Deno/TypeScript)
│       ├── modules-full/            GET  /api/v1/modules/full
│       ├── car-types/               GET  /api/v1/car-types
│       ├── industry-types/          GET  /api/v1/industry-types
│       ├── car-types-suggest/       POST /api/v1/car-types/suggest
│       └── modules-validate-name/   POST /api/v1/modules/validate-name
├── web/                         # Owner portal — Next.js + Tailwind frontend
│   ├── src/app/                     # Pages: /, /login, /register, /dashboard,
│   │                                #   /forgot-password, /reset-password
│   └── src/lib/supabase/            # Browser/server Supabase clients + session proxy
├── openapi/
│   └── modulerepo-openapi-v1.yaml   # REST contract for Free Dispatcher
├── docs/
│   ├── SUPABASE_SETUP.md            # Milestone 1 setup + verification guide
│   └── M2_API_TEST_PLAN.md          # Milestone 2 API test plan (curl/Postman)
├── .vscode/                     # Recommended settings & extensions
├── .env.example                 # Copy to .env and fill in (never commit .env)
└── .gitignore
```

## Prerequisites

- [VS Code](https://code.visualstudio.com/)
- [Supabase CLI](https://supabase.com/docs/guides/cli) — `brew install supabase/tap/supabase`
- [Deno](https://deno.land/) — for editing/testing Edge Functions locally
- Recommended VS Code extensions are listed in `.vscode/extensions.json`
  (VS Code will prompt you to install them when you open this folder)

## Getting started

```bash
# 1. Install the recommended extensions when prompted, or run:
code --install-extension denoland.vscode-deno

# 2. Authenticate and link to the dev project
supabase login
supabase link --project-ref dpifxkipqfaxujidgjyg

# 3. Copy environment template and fill in your keys
cp .env.example .env
#   (Dashboard -> Settings -> API for SUPABASE_ANON_KEY / SUPABASE_SERVICE_ROLE_KEY)

# 4. Apply all migrations to the linked project
supabase db push

# 5. Deploy the Edge Functions
supabase functions deploy modules-full
supabase functions deploy car-types
supabase functions deploy industry-types
supabase functions deploy car-types-suggest
supabase functions deploy modules-validate-name

# 6. (Optional) run everything locally first
supabase start            # spins up local Postgres, Studio, Auth, Storage
supabase db reset         # applies migrations/ to the local DB
supabase functions serve  # serves all functions locally with hot reload
```

## Local development loop

```bash
supabase start
supabase functions serve --env-file .env
# Edge Functions are now live at http://localhost:54321/functions/v1/...
# Studio (DB browser) at http://localhost:54323
```

Edit a migration → `supabase db reset` to re-apply from scratch locally,
or add a new timestamped file to `supabase/migrations/` for incremental changes
and run `supabase db push` against the linked project when ready.

## Frontend (Owner Portal)

`web/` is a Next.js + Tailwind app providing email/password auth — registration,
login, logout, and password reset — backed directly by Supabase Auth.

```bash
cd web
npm install
cp .env.local.example .env.local
#   fill in NEXT_PUBLIC_SUPABASE_URL / NEXT_PUBLIC_SUPABASE_ANON_KEY
#   (Dashboard -> Settings -> API, or `supabase status` for the local stack)
npm run dev
# http://localhost:3000
```

Role assignment is handled server-side: a `handle_new_user` trigger
(migration `...000009_handle_new_user.sql`) creates the matching
`owner_profiles` row on signup — the very first account in the project
becomes `admin`, every account after that is an `owner`. Clients can never
self-assign a role.

## Testing the API

Follow `docs/M2_API_TEST_PLAN.md` for curl scripts covering RLS behavior
(anonymous / owner / admin) and all five Edge Functions. You can also import
`openapi/modulerepo-openapi-v1.yaml` directly into Postman or the OpenAPI
VS Code extension for an interactive contract view.

## Architecture notes

- **ADR-001**: The boundary between Module Repository and Free Dispatcher is
  one-directional and read-only. `GET /api/v1/modules/full` is the single
  integration point — never violate this boundary with writes from Dispatcher.
- **Record numbers**: `freemon_modules.record_number` follows `FMN-XXXX`,
  auto-generated by a sequence + trigger (see migration `...000005_triggers.sql`).
- **RLS**: every table is governed by Row-Level Security — see migration
  `...000006_rls.sql` for the full policy matrix (public / owner / show_master / admin).
